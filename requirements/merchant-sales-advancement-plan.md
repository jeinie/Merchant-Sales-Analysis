# 가맹점 매출 분석 플랫폼 고도화 기획서

## 1. 기획 목적

현재 프로젝트는 지도 기반 가맹점 매출 모니터링, 역할 기반 접근 제어, 월별 매출 추이, 업종/지역 평균 비교, Gemini 기반 AI 운영 인사이트를 제공한다. 고도화의 목적은 이 기능을 단순 조회형 대시보드에서 한 단계 확장해, 본사 관리자와 영업 담당자가 "어느 매장을 먼저 확인해야 하는지", "왜 확인해야 하는지", "조치 이후 상태가 개선되고 있는지"까지 추적할 수 있는 운영 관리 플랫폼으로 발전시키는 것이다.

핵심 방향은 다음과 같다.

- 매출 변화 감지를 자동화해 담당자가 놓치는 지점을 줄인다.
- AI 분석 결과를 일회성 텍스트가 아니라 저장, 비교, 이력 관리 가능한 운영 자산으로 만든다.
- 관리자 페이지를 단순 권한 설정 화면에서 담당자 업무 배정 및 성과 추적 화면으로 확장한다.
- 데이터 수집, 검증, 배포, 보안 구조를 정리해 실제 운영 환경으로 이전하기 쉬운 형태를 만든다.

## 2. 현재 상태 요약

### 2.1 구현된 주요 기능

- JWT 기반 로그인 및 사용자 인증
- 관리자/영업사원 역할 기반 데이터 접근 제어
- 영업사원별 담당 가맹점 조회 범위 제한
- Kakao Maps 기반 가맹점 위치 시각화
- 최근 월 매출 규모에 따른 지도 마커 크기 차등 표시
- 전월 대비 성장률 기준 상승/보합/하락 마커 색상 구분
- 지역/업종 필터링 및 가맹점 목록 선택
- 월별 매출 추이 차트 및 업종 평균 비교
- 관리자 페이지의 가맹점 담당자 배정
- 영업사원별 AI 분석 권한 ON/OFF
- Gemini 기반 AI 운영 인사이트 생성 및 모달 확대 보기

### 2.2 현재 데이터 구조

현재 핵심 테이블은 다음과 같다.

- `users`: 사용자, 역할, AI 사용 권한
- `merchants`: 가맹점 기본 정보, 주소, 좌표
- `monthly_sales`: 가맹점별 월 매출, 거래 건수, 객단가
- `user_merchant_assignments`: 영업사원과 가맹점 배정 관계

현재 구조는 MVP 분석에는 충분하지만, 알림 이력, 조치 이력, AI 분석 저장, 목표 관리, 데이터 업로드 검증을 처리하기에는 별도 테이블이 필요하다.

## 3. 고도화 목표

### 3.1 운영 우선순위 자동화

대시보드를 열었을 때 담당자가 직접 모든 가맹점을 훑지 않아도, 시스템이 점검이 필요한 가맹점을 우선순위로 정렬한다.

- 급락/급등 가맹점 자동 감지
- 업종 평균 대비 저성과 가맹점 식별
- 지역 평균 대비 이탈 매장 식별
- 거래 건수 감소와 객단가 감소 원인 분리
- "정상 관찰", "주의", "점검 필요" 등급 산정

### 3.2 AI 인사이트 운영화

현재 AI 분석은 화면에서 버튼을 눌러 즉시 생성하는 형태다. 고도화 후에는 분석 결과를 저장하고, 이전 분석과 비교하며, 담당자가 후속 조치를 기록할 수 있어야 한다.

- AI 분석 결과 저장
- 분석 생성 시점, 생성 사용자, 대상 가맹점 기록
- 이전 AI 인사이트 조회
- 동일 가맹점의 인사이트 변화 비교
- AI 인사이트 기반 후속 조치 등록

### 3.3 관리자 업무 관리 확장

관리자는 담당자 배정뿐 아니라 담당자별 관리 대상, 점검 필요 매장, 조치 완료 여부를 확인할 수 있어야 한다.

- 담당자별 배정 가맹점 수 확인
- 담당자별 점검 필요 매장 수 확인
- 미처리 알림/조치 건수 표시
- 담당자 변경 이력 조회
- AI 권한 외 추가 권한 관리 구조 확장

### 3.4 데이터 신뢰성 강화

월별 매출 데이터가 잘못 들어오면 모든 분석 결과가 흔들린다. 운영 단계에서는 데이터 입력/업로드 검증, 결측치 처리, 이상값 플래그가 필요하다.

- 월별 매출 업로드 기능
- 업로드 전 데이터 검증
- 중복 월 데이터 처리 정책
- 결측 좌표/주소 검증
- 전월 대비 과도한 변동 자동 플래그

## 4. 고도화 범위

### 4.1 1차 고도화: 알림 및 우선순위

목표는 현재 화면에서 가장 빠르게 체감할 수 있는 운영 기능을 추가하는 것이다.

#### 기능 요구사항

- 전월 대비 매출 하락률이 임계치 이하인 가맹점 알림 생성
- 전월 대비 매출 상승률이 임계치 이상인 가맹점 알림 생성
- 거래 건수와 객단가 중 어떤 지표가 변화에 더 크게 기여했는지 표시
- 지도/목록/상세 패널에서 동일한 위험 등급 표시
- 알림 클릭 시 해당 가맹점 상세 패널로 이동
- 관리자가 알림 기준값을 설정할 수 있는 구조 준비

#### 권장 기본 기준

- 매출 하락 주의: 전월 대비 -10% 이하
- 매출 하락 점검 필요: 전월 대비 -15% 이하
- 매출 급등 확인 필요: 전월 대비 +20% 이상
- 거래 건수 급감: 전월 대비 -12% 이하
- 객단가 급감: 전월 대비 -10% 이하

#### 백엔드 변경안

- `GET /api/alerts` 추가
- `GET /api/merchants` 응답에 `riskLevel`, `priorityScore`, `alertTags` 추가 검토
- 우선순위 계산 로직을 서비스 계층으로 분리

#### 프론트엔드 변경안

- 헤더 알림 팝오버를 API 기반으로 변경
- 사이드바 목록에 위험 등급 배지 추가
- 지도 마커 선택 시 알림 요약 표시
- 상세 패널 상단에 "점검 사유" 영역 추가

### 4.2 2차 고도화: AI 인사이트 저장 및 이력

목표는 AI 분석을 일회성 호출에서 이력 기반 운영 도구로 전환하는 것이다.

#### 기능 요구사항

- AI 분석 생성 시 결과를 DB에 저장
- 최근 분석 결과를 상세 패널에서 자동 표시
- "새 분석 생성"과 "이전 분석 보기" 분리
- AI 분석 결과에 운영 태그, 위험 등급, 요약 문장 저장
- 담당자가 분석 결과에 메모를 남길 수 있음

#### 신규 테이블 초안

```sql
CREATE TABLE ai_insight_histories (
    id BIGINT NOT NULL AUTO_INCREMENT,
    merchant_id VARCHAR(20) NOT NULL,
    created_by VARCHAR(64) NOT NULL,
    sales_month CHAR(7) NOT NULL,
    risk_level VARCHAR(20) NOT NULL,
    summary VARCHAR(500) NOT NULL,
    content TEXT NOT NULL,
    tags VARCHAR(255),
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (id),
    KEY idx_ai_insight_merchant_month (merchant_id, sales_month),
    CONSTRAINT fk_ai_insight_merchant
        FOREIGN KEY (merchant_id) REFERENCES merchants (id)
        ON DELETE CASCADE,
    CONSTRAINT fk_ai_insight_user
        FOREIGN KEY (created_by) REFERENCES users (id)
        ON DELETE CASCADE
);
```

#### API 변경안

- `GET /api/merchants/{id}/ai-insights`
- `POST /api/merchants/{id}/ai-insights`
- `GET /api/merchants/{id}/ai-insights/latest`

#### 프론트엔드 변경안

- 상세 패널에 최근 AI 분석 카드 표시
- AI 인사이트 모달에 이력 탭 추가
- 생성 중 상태, 실패 상태, 저장 실패 상태 분리

### 4.3 3차 고도화: 담당자 조치 관리

목표는 알림을 확인하는 데서 끝나지 않고, 담당자가 어떤 후속 조치를 했는지 기록하고 관리자가 진행 상황을 볼 수 있게 하는 것이다.

#### 기능 요구사항

- 알림 또는 AI 인사이트에서 조치 항목 생성
- 조치 상태 관리: `OPEN`, `IN_PROGRESS`, `DONE`, `DISMISSED`
- 담당자 메모 등록
- 처리 기한 설정
- 관리자 페이지에서 미처리 조치 목록 조회
- 담당자별 처리 현황 요약

#### 신규 테이블 초안

```sql
CREATE TABLE action_items (
    id BIGINT NOT NULL AUTO_INCREMENT,
    merchant_id VARCHAR(20) NOT NULL,
    assigned_user_id VARCHAR(64),
    source_type VARCHAR(30) NOT NULL,
    source_id BIGINT,
    title VARCHAR(200) NOT NULL,
    description TEXT,
    status VARCHAR(30) NOT NULL DEFAULT 'OPEN',
    due_date DATE,
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    PRIMARY KEY (id),
    KEY idx_action_merchant_status (merchant_id, status),
    KEY idx_action_assignee_status (assigned_user_id, status),
    CONSTRAINT fk_action_merchant
        FOREIGN KEY (merchant_id) REFERENCES merchants (id)
        ON DELETE CASCADE,
    CONSTRAINT fk_action_assignee
        FOREIGN KEY (assigned_user_id) REFERENCES users (id)
        ON DELETE SET NULL
);
```

#### 화면 변경안

- 상세 패널에 "조치 등록" 버튼 추가
- 관리자 페이지에 조치 현황 탭 추가
- 영업사원 대시보드에 "내 미처리 조치" 영역 추가

### 4.4 4차 고도화: 매출 데이터 업로드 및 검증

목표는 seed SQL 중심의 데이터 관리에서 운영자가 월별 매출 데이터를 갱신할 수 있는 구조로 확장하는 것이다.

#### 기능 요구사항

- CSV 업로드
- 업로드 미리보기
- 필수 컬럼 검증: 가맹점 ID, 월, 매출, 거래 건수, 객단가
- 존재하지 않는 가맹점 ID 검출
- 동일 가맹점/동일 월 데이터 중복 검출
- 덮어쓰기/건너뛰기 정책 선택
- 업로드 결과 요약 제공

#### API 변경안

- `POST /api/admin/sales-upload/preview`
- `POST /api/admin/sales-upload/commit`
- `GET /api/admin/sales-upload/histories`

#### 운영 규칙

- 음수 매출, 음수 거래 건수는 업로드 차단
- 거래 건수가 0인데 매출이 0보다 크면 검토 필요로 표시
- 객단가는 업로드 값과 `sales / tx_count` 계산값 차이가 클 경우 경고
- 전월 대비 +-50% 이상 변동은 이상값 후보로 표시

## 5. 비기능 요구사항

### 5.1 보안

- 운영 환경에서 JWT secret 기본값 사용 금지
- 관리자 API는 서버에서 역할 검증 필수
- AI API 키는 프론트엔드 노출 구조에서 백엔드 프록시 구조로 이전 검토
- CORS `*` 설정을 운영 도메인 기반으로 제한
- 로그인 실패 횟수 제한 및 계정 잠금 정책 검토

### 5.2 성능

- 가맹점 수 증가에 대비해 월별 매출 조회 쿼리 인덱스 유지

- 평균값 계산은 매 요청마다 전체 데이터를 순회하지 않도록 캐시 또는 집계 테이블 검토
- AI 인사이트는 동일 월/동일 가맹점에 대해 재사용 가능한 저장 구조 적용

### 5.3 운영 및 관측성

- API 오류 로그 구조화
- AI 호출 실패율 및 응답 시간 기록
- DB 연결 실패, 인증 실패, 업로드 실패 로그 분리
- 운영 배포 시 환경변수 점검 체크리스트 제공

### 5.4 테스트

- 우선순위 계산 단위 테스트
- 역할별 API 접근 제어 테스트
- 담당자 배정 변경 테스트
- AI 인사이트 저장/조회 테스트
- CSV 업로드 검증 테스트
- 프론트엔드 주요 흐름 빌드 및 렌더링 검증

## 6. 우선순위 로드맵

### Phase 1: 운영 알림 MVP

- 위험 등급 계산 로직 추가
- `/api/alerts` API 추가
- 헤더 알림 팝오버 API 연동
- 상세 패널 점검 사유 표시
- 위험 등급 단위 테스트 추가

완료 기준:
- 관리자가 로그인했을 때 전체 가맹점의 점검 필요 알림을 확인할 수 있다.
- 영업사원은 본인 담당 가맹점의 알림만 볼 수 있다.
- 알림 클릭 시 해당 가맹점 상세 정보로 이동한다.

### Phase 2: AI 인사이트 이력화

- `ai_insight_histories` 테이블 추가
- AI 생성 API를 백엔드로 이전
- AI 분석 결과 저장/조회 API 추가
- 상세 패널에 최근 분석 및 이력 모달 추가

완료 기준:
- 같은 가맹점을 다시 열어도 최근 AI 분석 결과를 볼 수 있다.
- 분석 생성자와 생성 시점이 기록된다.
- AI 권한이 없는 사용자는 생성 API를 호출할 수 없다.

### Phase 3: 조치 관리

- `action_items` 테이블 추가
- 조치 등록/상태 변경 API 추가
- 관리자 페이지 조치 현황 탭 추가
- 영업사원별 미처리 조치 표시

완료 기준:
- 점검 필요 알림에서 조치 항목을 만들 수 있다.
- 담당자는 조치 상태를 변경하고 메모를 남길 수 있다.
- 관리자는 담당자별 미처리 현황을 볼 수 있다.

### Phase 4: 데이터 업로드

- CSV 업로드 미리보기 API 추가
- 검증 결과 UI 추가
- 업로드 확정 및 이력 저장
- 이상값 후보 표시

현재 구현 상태:

- `POST /api/admin/sales-upload/preview`로 CSV 업로드 전 미리보기 검증을 수행한다.
- `POST /api/admin/sales-upload/commit`으로 오류가 없는 CSV를 `monthly_sales`에 반영한다.
- `GET /api/admin/sales-upload/histories`로 최근 매출 데이터 반영 이력을 확인한다.
- 관리자 페이지의 `데이터 업로드` 탭에서 CSV 파일을 선택하고 행별 오류/경고를 확인할 수 있다.
- 필수 헤더, 존재하지 않는 가맹점, 동일 파일 내 중복, 기존 월 데이터, 음수값, 객단가 불일치, 전월 대비 50% 이상 변동을 검증한다.
- CSV 반영은 가맹점/매출월 기준으로 upsert하며, 반영 이력에 파일명, 업로드 사용자, 행 수, 경고 수, 반영 시각을 남긴다.

완료 기준:
- 관리자는 월별 매출 CSV를 업로드하기 전에 오류/경고를 확인할 수 있다.
- 정상 데이터만 반영하거나 중복 처리 정책을 선택할 수 있다.
- 업로드 결과가 이력으로 남는다.

## 7. 화면별 개선안

### 7.1 대시보드

- 요약 카드에 "점검 필요", "주의", "급등 확인" 건수 추가
- 알림 목록을 위험도순으로 정렬
- 필터 적용 시 알림 수와 지도 마커가 함께 갱신
- 매출 규모뿐 아니라 위험 등급 기준 정렬 옵션 추가

### 7.2 지도

- 선택한 위험 등급만 지도에 표시하는 토글 추가
- 마커 인포윈도우에 점검 사유 1줄 표시
- 좌표 미확인 매장을 별도 목록으로 제공

### 7.3 상세 패널

- 상단에 위험 등급, 우선순위 점수, 점검 사유 표시
- 거래 건수/객단가 증감률을 별도 미니 지표로 표시
- AI 인사이트 이력과 조치 항목을 탭으로 분리

### 7.4 관리자 페이지

- "담당자 배정", "권한 관리", "조치 현황", "데이터 업로드" 탭 구조로 확장
- 담당자별 배정 수, 점검 필요 수, 미처리 조치 수 요약
- 담당자 변경 이력 표시

## 8. 데이터 모델 확장 요약

고도화를 위해 추가 검토할 테이블은 다음과 같다.

- `alert_rules`: 알림 기준값 관리
- `alert_events`: 생성된 알림 이력
- `ai_insight_histories`: AI 분석 결과 저장
- `action_items`: 담당자 후속 조치 관리
- `sales_upload_histories`: 매출 데이터 업로드 이력
- `assignment_histories`: 담당자 배정 변경 이력

테이블 추가 순서는 `alert_events`, `ai_insight_histories`, `action_items`, `sales_upload_histories` 순서를 권장한다. 사용자가 체감하는 가치가 크고, 기존 구조와 결합하기 쉬운 순서이기 때문이다.

## 9. 리스크 및 대응 방안

| 리스크 | 영향 | 대응 |
|---|---|---|
| AI API 키가 프론트엔드에 노출됨 | 운영 보안 취약 | AI 호출을 백엔드 API로 이전 |
| 평균값 계산이 요청마다 반복됨 | 데이터 증가 시 응답 지연 | 집계 캐시 또는 월별 집계 테이블 검토 |
| 알림 기준이 하드코딩됨 | 운영 정책 변경이 어려움 | 관리자 설정 테이블로 분리 |
| 담당자 배정 이력이 없음 | 운영 추적 어려움 | `assignment_histories` 추가 |
| 잘못된 매출 데이터 입력 | 분석 신뢰도 하락 | 업로드 검증 및 이상값 플래그 적용 |

## 10. 성공 지표

- 점검 필요 가맹점 식별 시간 감소
- 담당자별 미처리 조치 건수 추적 가능
- 동일 가맹점 AI 인사이트 재생성 횟수 감소
- 월별 매출 데이터 업로드 오류 사전 차단
- 관리자/영업사원 역할별 접근 제어 테스트 통과

## 11. 추천 구현 순서

1. 위험 등급 계산 로직을 백엔드 서비스로 분리한다.
2. 알림 API와 프론트엔드 알림 UI를 실제 API 기반으로 연결한다.
3. AI 호출을 백엔드로 이전하고 분석 결과를 DB에 저장한다.
4. 상세 패널에 AI 이력과 조치 등록 기능을 추가한다.
5. 관리자 페이지를 탭 구조로 정리하고 조치 현황을 추가한다.
6. CSV 업로드와 데이터 검증 기능을 추가한다.
7. 배포 전 보안 설정, CORS, JWT secret, API 키 관리 방식을 정리한다.
